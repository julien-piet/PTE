HOW THESE TASKS ARE EVALUATED (program_html)
--------------------------------------------------------------------------------

The eval does NOT just check the API response. It:
  1. Runs the agent (API calls, note posted)
  2. Opens a fresh Playwright browser session, logs in to GitLab
  3. Navigates to the hardcoded eval URL (a specific MR page)
  4. Extracts content using a JS locator from the DOM
  5. Checks the extracted text against required_contents

This is important: the note must BOTH be posted via API AND be visible in the
live GitLab web UI for the eval to pass.

--- [390] PERSISTENT FAILURE (all 3 runs) — Root cause: DOM selector/lastElementChild ---

The note IS posted correctly to MR 1531 in all 3 runs (API confirms success).
The eval navigates to MR 1531 and uses:
  lastElementChild of notes-list → .timeline-discussion-body → outerText
  exact_match: "lgtm"

This has ALWAYS failed (all 3 runs). Root cause options:

  Option A — lastElementChild is a system note:
    GitLab MRs have activity events (pipeline status, branch pushes, reviewer
    assignments, etc.) mixed into the notes-list alongside user comments.
    In a fresh Docker instance, these system notes are created when the fixture
    is loaded. If a system note has a later timestamp than the user comment,
    it becomes lastElementChild and has no .timeline-discussion-body → TypeError
    → locator fails → eval fails.

  Option B — CSS selector is malformed:
    The locator string contains `[id="notes-list"` with no closing `]`.
    In strict CSS parsers this is a SyntaxError. Playwright propagates this as
    a page.evaluate() exception → locator fails → eval fails.
    However, Chromium may parse this leniently and accept it, given that 391
    and 392 appear to have passed in run A using the same selector.

  Option C — "lgtm" note text in GitLab renders differently:
    The note body is "lgtm" but the UI might surround it with extra text or
    GitLab might interpret it as something else. Unlikely.

Most likely: A combination of Option A (system notes) and the fresh Docker
instances in B/C not having pre-existing state.

Fix needed (eval infrastructure): Change the locator to find the LAST USER
COMMENT, not the last element in notes-list. Something like:
  querySelectorAll('.timeline-entry:not(.system-note)') → last item → .body.outerText
Or use CSS selector `:not(.system-note)` to skip system events.

--------------------------------------------------------------------------------
CROSS-CUTTING ROOT CAUSES (what actually needs to change)
--------------------------------------------------------------------------------

1. EVAL INFRASTRUCTURE BUG — system notes as lastElementChild (affects 390, 391)

   The JS locator `notes-list lastElementChild .timeline-discussion-body` is
   fragile — it assumes the last item in the notes list is always a user comment.
   In fresh GitLab Docker instances, MR activity events (system notes) may have
   timestamps AFTER user notes and appear as lastElementChild.

   Fix: Update the locator to `.timeline-entry:not(.system-note) + ...` or use
   `querySelectorAll` to find the last non-system note. Alternatively, look for
   the note by body content instead of by position.

2. EVAL INFRASTRUCTURE BUG — async comment rendering not waited for (affects 389, C)

   The evaluator waits for notes-list only when the locator string contains
   "notes-list". For empty locators (task 389), it calls `document.body.outerText`
   immediately after networkidle, before Vue.js renders comments.

   Fix: Either always wait for notes-list, or add an explicit wait in the
   evaluator for MR comment tasks with empty locators.

3. PLANNER BUG (run B, task 389) — hardcoded wrong MR iid

   B's planner generated step_1 as GET notes for `merge_request_iid="1"` (literal
   "1"). This is a nonsensical plan. C's planner generates a correct 3-step plan
   for the same task. This specific B regression appears resolved in C.

4. SEARCH RETURNS WRONG MR (task 392) — search term doesn't match MR title well

   `search="color"` → first result is "update or remove 404 links" (iid=1485)
   `search="color utility"` → first result is "Add a11y-contrast tool" (iid=1145)
   Correct MR is at position [1] when searching "color utility".

   Fix: The planner should either:
     a. Use the more specific search term "color utility classes", OR
     b. Not blindly pick result[0] — look for best title match, OR
     c. Use a narrower search that eliminates the false positive

5. MR SEARCH RETURNS EMPTY (task 393) — title mismatch between task description and actual title

   "fixing the broken links" doesn't match any MR title in byteblaze/empathy-prompts.
   MR 19 exists but has a different title. The agent needs a fallback: list all
   open MRs (no search filter) and pick the most relevant one.

   Fix: When search returns empty, retry with a broader query or paginate through
   all MRs in the repo instead of returning an error.
